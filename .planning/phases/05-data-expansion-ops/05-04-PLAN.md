`.github/workflows/db-backup.yml`을 아래 내용으로 생성한다.

핵심 요구사항:
- cron: `'0 19 * * 0'` — 일요일 19:00 UTC = 월요일 04:00 KST
- `workflow_dispatch: {}` — 수동 테스트 가능
- `timeout-minutes: 15`
- `apt-get install postgresql-client` — pg_dump 실행 환경
- `SUPABASE_DB_URL` — secrets 참조. **절대 echo나 출력 금지**
- `BACKUP_PAT` — secrets 참조
- 파일명: `backup-$(date +%Y-%m-%d).sql.gz`
- backup repo: `nickujung-art/danjiondo-backup`
- 90일 cleanup: `date -d '90 days ago'`로 CUTOFF 계산 후 이전 파일 git rm

```yaml
name: DB Backup

on:
  schedule:
    - cron: '0 19 * * 0'   # 일요일 19:00 UTC = 월요일 04:00 KST
  workflow_dispatch: {}

jobs:
  backup:
    name: pg_dump → danjiondo-backup repo
    runs-on: ubuntu-latest
    timeout-minutes: 15

    steps:
      - name: Install postgresql-client
        run: |
          sudo apt-get update -qq
          sudo apt-get install -y postgresql-client

      - name: Run pg_dump
        env:
          SUPABASE_DB_URL: ${{ secrets.SUPABASE_DB_URL }}
        run: |
          set -e
          FILENAME="backup-$(date +%Y-%m-%d).sql.gz"
          pg_dump "$SUPABASE_DB_URL" | gzip > "/tmp/$FILENAME"
          echo "FILENAME=$FILENAME" >> "$GITHUB_ENV"
          echo "pg_dump complete: $FILENAME"

      - name: Clone backup repo
        env:
          BACKUP_PAT: ${{ secrets.BACKUP_PAT }}
        run: |
          set -e
          git clone \
            "https://x-access-token:${BACKUP_PAT}@github.com/nickujung-art/danjiondo-backup.git" \
            /tmp/backup-repo

      - name: Commit backup file
        run: |
          set -e
          cp "/tmp/$FILENAME" /tmp/backup-repo/
          cd /tmp/backup-repo
          git config user.email "github-actions[bot]@users.noreply.github.com"
          git config user.name "github-actions[bot]"
          git add "$FILENAME"
          git commit -m "backup: $FILENAME"
          git push

      - name: Cleanup backups older than 90 days
        run: |
          set -e
          cd /tmp/backup-repo
          CUTOFF=$(date -d '90 days ago' +%Y-%m-%d)
          REMOVED=0
          for f in backup-*.sql.gz; do
            [ -f "$f" ] || continue
            FILEDATE="${f#backup-}"
            FILEDATE="${FILEDATE%.sql.gz}"
            if [[ "$FILEDATE" < "$CUTOFF" ]]; then
              git rm "$f"
              REMOVED=$((REMOVED + 1))
            fi
          done
          if [ "$REMOVED" -gt 0 ]; then
            git commit -m "cleanup: remove $REMOVED backup(s) older than 90 days"
            git push
            echo "Removed $REMOVED old backup(s)"
          else
            echo "No old backups to remove"
          fi
```

**보안 주의사항 (코드에 포함 안 됨 — 운영자 공지용):**
- `SUPABASE_DB_URL`은 Connection string에 패스워드를 포함한다. `echo "$SUPABASE_DB_URL"` 절대 금지.
- `set -e`는 명령 실패 시 즉시 종료. URL 노출 없이 에러 핸들링 가능.
- PAT은 `repo` scope — danjiondo-backup 레포 외에도 접근 가능. Fine-grained PAT(Contents write만)으로 교체 권장.

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-04-01 | Information Disclosure | SUPABASE_DB_URL (Connection URL에 패스워드 포함) | mitigate | GitHub Secrets에만 저장. workflow 내 `echo`, `cat`, `print` 절대 금지. `set -e`로 에러 처리. GitHub Actions는 Secrets를 자동 마스킹 |
| T-05-04-02 | Information Disclosure | BACKUP_PAT scope 과잉 (repo 전체) | mitigate | 현재: Classic PAT repo scope. 권고: Fine-grained PAT으로 교체 — Contents write 권한을 danjiondo-backup repo로만 제한 (Phase 7 이전 개선 권장) |

| T-05-04-04 | Tampering | pg_dump 실패 시 빈 파일 커밋 | mitigate | `set -e` — pg_dump 실패 시 스크립트 즉시 종료. 빈 파일 커밋 없음 |
